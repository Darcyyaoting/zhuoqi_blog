Crontab调用python错误

问题：crontab调用python时出现ImportError: No module named XXX， 但是shell里单独执行是OK的

解决方法：
因为crontab 默认找 python 的默认路径，所以需要指定 python 路径
首先，which python，确定python的路径
然后把整个python的路径都复制进crontab里去

图解Http7

第七章 确保web安全的https

  1. http的缺点

    1.1 通信使用明文（不加密），内容可能被窃听；
    抓包工具：wireshark等
    应对措施：加密处理
    第一种是通信加密：http中没有加密机制，但可以和ssl（secure socket layer）或者tls（transport layer security）的组合使用，加密http的通内容
    第二种是内容加密
    1.2 不验证通信方的内容，有可能遭遇伪装
    应对措施：证书
    证书由值得信赖的第三方机构颁发，客户端发送请求之前先向第三方机构确认服务端证书
    1.3 无法证明报文的完整性，有可能被篡改
    请求或响应在传输途中被篡改的攻击叫中间人攻击（man-in-the-middle attack,MITM）

  2. http+加密+证书+完整性保护 = https

    https是通信接口部分用ssl和tls代替的http。通常http和tcp直接通信，使用ssl时，变成http首先和ssl通信，然后ssl和tcp通信


    ssl 是独立于http的协议，其他运行在应用层的协议如：SMTP、Telnet等都可以配合ssl使用。ssl是目前应用最为广泛的网络安全技术。
    2.1 加密技术
    加密方法中加密算法都是公开的，密钥是私有的，加密和解密都要用到密钥。加密和解密使用同一份密钥叫做共享密钥加密，也被叫做对称密钥加密。但是在传输密钥时如果被窃听，则失去了密钥的意义。
    公开密钥加密：使用公开密钥和私有密钥，所以是非对称密钥加密。使用公开密钥加密，使用私有密钥解密通信。
    https使用公开密钥和共享密钥混合的加密方式。使用公开密钥加密传输密钥，然后使用共享密钥加密传输报文。因为公开密钥加密比共享密钥加密要复杂的多，也消耗更多的时间。
    2.2 证书
    公开密钥如果被篡改，不是原先服务器的公开密钥，那么会有危险。需要使用由数字证书认证机构颁发的公开密钥证书。

图解Http4

第四章 返回结果的HTTP状态码

  1. 状态码告知从服务器端返回的请求结果

    状态码由3位数字和原因短语组成



  2. 2xx成功

    2xx的响应结果表明请求被正常处理了
    2.1 200 ok
    表示从客户端发来的请求在服务端被正常处理了
    2.2 204 no content
    表示请求已成功处理，但在返回的响应报文中不含实体的主体部分，也不允许返回任何实体的主体。
    2.3 206 partial content
    表示客户端进行了范围请求，响应报文中包含由content-range指定范围的实体内容
  3. 3xx重定向
  4. 4xx客户端错误

    表明客户端是发生错误的原因所在
    4.1 400 bad request
    表示请求报文中存在语法错误
    4.2 401 unauthorized
    表示发送的请求需要有通过HTTP认证（BASIC 认证、DIGEST认证）的认证信息。若之前进行过1次请求，则表示用户认证失败。
    4.3 403 forbidden
    表明请求资源的访问被服务器拒绝了。
    4.4 404 not found
    表明服务器上无法找到请求的资源。
  5. 5xx服务器错误

    表明服务器端本身发生错误
    5.1 500 internal server error
    表明服务器端在执行请求时发生错误
    5.2 503 service unavailable
    表明服务器暂时处于超负载或正在进行停机维护

图解Http3

第三章 HTTP报文内的HTTP信息

  1. HTTP报文

    HTTP报文由报文首部和报文主体构成，通常并不一定有报文主体。请求端（客户端）发送的http报文叫做请求报文，响应端（服务器端）发送的http报文叫做响应报文。
  2. 请求报文及响应报文的结构

    报文首部：请求行/状态行、首部字段（请求首部字段/响应首部字段、通用首部字段和实体首部字段）、其他（例如：cookie）。
  3. 编码提升传输速率

    3.1 报文（message）：http通信的基本单位，由8位组字节流组成，通过http通信传输；实体（entity）：作为请求或响应的有效载荷数据被传输，其内容由实体首部和实体主体组成。
    http报文的主体用来传输实体主体，通常报文主体等于实体主体，只有当传输中进行编码操作时，实体主体内容发生变化，才导致和报文主体产生差异。
    3.2 http内容编码：指明应用在实体内容上的编码格式，并保持试题信息原样压缩。内容编码后的实体由客户端接收并负责解码。
    常用内容编码：gzip（GNU zip）、compress（unix系统的标准压缩）、deflate（zlib）、identity（不进行编码）
    3.3 分割发送的分块传输编码
    在传输大容量数据时，通过把数据分割成多块，能够让浏览器逐步显示页面，这种把实体主体分块的功能叫分块传输编码。
  4. 发送多种数据的多部分对象集合

    MIME（multipurpose internet mail extensions）允许邮件处理文本、图片、视频等多个不同类型的数据。在MIME扩展中会使用一种multipart的方法，来容纳多份不同类型的数据。
    multipart/form-data 在web表单文件上传时使用；multipart/byteranges 状态码206响应报文包含了多个范围的内容时使用。
  5. 内容协商

    当浏览器的默认语言为英文或中文时，访问相同的uri的web页面时，会显示对应的英文版或中文版的web页面，这样的机制成为内容协商。
    分类：服务器驱动协商（参考请求的首部字段）、客户端驱动协商（用户手动选择）、透明协商（服务器驱动和客户端驱动的结合体）

图解Http2

第二章 简单的http协议

http请求和响应

  1. 请求报文由以下几部分构成：请求方法、请求URI、协议版本、可选的请求首部字段和内容实体



  2. 响应报文由以下几部分构成：协议版本、状态码（表示请求成功或失败的数字代码）、用以解释状态码的原因短语、
    可选的响应首部字段以及实体主体



  3. http是一种无状态协议，对请求和响应之间的通信状态不进行保存。所以引进了cookie技术。

  4. 请求URI
    GET http://hackr.jp/index.htm HTTP/1.1 或者
    GET /index.htm HTTP/1.1
    Host: hackr.jp
    如果不是访问特定资源而是对服务器本身发起请求，可以用*代替请求URI，比如：
    OPTIONS * HTTP/1.1

  5. HTTP方法
    GET：用来请求访问已被URI识别的资源。
    请求：GET /index.html HTTP/1.1
    Host: www.hackr.jp
    响应：返回 index.html 的页面资源
    POST：传输实体主体
    请求：POST /submit.cgi HTTP/1.1
    Host: www.hackr.jp
    Content-Length: 1560（1560字节的数据）
    响应： 返回submit.cgi接收数据的处理结果
    PUT：传输文件
    像FTP协议的文件上传一样，要求在请求报文的主体中包含文件内容，然后保存到请求URI指定的位置。但是因为不带验证机制，一般的web网站不使用此方法（rest标准的web网站可能开放）
    DELETE：删除文件

  6. 持久连接
    http早期采用每完成一次请求之后，就断开tcp一次，即tcp三次握手四次挥手，这样会浪费很多tcp连接时间。后来http1.1改为持久连接，即如果一方不声明断开连接，则保持tcp连接状态

  7. 管线化
    一次http请求发送后不用等待回应可以发送下一次请求，即同时并行发送多个请求。

  8. cookie
    请求报文（没有cookie信息的状态）
    GET /reader/ HTTP/1.1
    Host: hackr.jp
    *首部字段内没有cookie的相关信息

    响应报文（服务器端生成cookie信息）
    HTTP/1.1 200 OK
    Date: Thu, 12 Jul 2012 07:12:20 GMT
    Server: Apache
    <Set-Cookie: sid=1342077140226724; path=/; expires=Wed, 10-Oct-12 07:12:20 GMT>
    Content-Type: text/plain; charset=UTF-8

    请求报文（自动发送保存着的cookie信息）
    GET /image/ HTTP/1.1
    Host: hackr.jp
    Cookie: sid=1342077140226724
